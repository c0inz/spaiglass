# Phase 2 Specification: Public Relay Server

## Overview

A public web server that acts as a Telegram-style relay between users' browsers
and their Claude WebUI instances running on private VMs. The relay never stores
API keys, project files, or conversation content. It is a pure authenticated
routing layer.

Users register once, upload a config file generated by their Claude agent, and
their VM becomes accessible from any browser via the public URL.

## Database Schema

SQLite. Three tables only.

```sql
CREATE TABLE users (
  id          TEXT PRIMARY KEY,           -- UUID
  github_id   TEXT UNIQUE NOT NULL,       -- GitHub user ID
  username    TEXT NOT NULL,              -- GitHub username
  email       TEXT,                       -- GitHub email (may be null if private)
  avatar_url  TEXT,                       -- GitHub avatar URL
  created_at  INTEGER NOT NULL,           -- Unix timestamp
  last_seen   INTEGER NOT NULL            -- Unix timestamp
);

CREATE TABLE connectors (
  id          TEXT PRIMARY KEY,           -- UUID, used as relay_token in config
  user_id     TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name        TEXT NOT NULL,              -- User-defined name e.g. "designer-01"
  vm_role     TEXT NOT NULL,              -- e.g. "Designer"
  created_at  INTEGER NOT NULL,
  last_connected INTEGER,                 -- Unix timestamp of last connection
  UNIQUE(user_id, name)
);

CREATE TABLE sessions (
  id          TEXT PRIMARY KEY,           -- Session token (in cookie)
  user_id     TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  created_at  INTEGER NOT NULL,
  expires_at  INTEGER NOT NULL,           -- Unix timestamp
  user_agent  TEXT,
  ip_address  TEXT
);
```

No conversation history. No file contents. No API keys. Ever.

## Authentication Flow

### GitHub OAuth

```
1. User visits /auth/github
2. Redirect to GitHub authorize URL with scopes: read:user, user:email
3. GitHub redirects to /auth/github/callback?code=...
4. Exchange code for access token via GitHub API
5. Fetch user profile from GitHub API
6. Upsert user record in database
7. Create session record, set signed session cookie
8. Redirect to /dashboard
```

**Routes:**

```
GET  /auth/github             Redirect to GitHub OAuth
GET  /auth/github/callback    Handle OAuth callback
POST /auth/logout             Delete session, clear cookie
GET  /api/auth/me             Return current user { id, username, avatarUrl }
```

Session cookie: `cwui_relay_session`, httpOnly, secure, sameSite: strict, 30 day expiry.

## Connector Management

### Registration

```
POST /api/connectors
```

Auth: required (session cookie)
Body:

```json
{
  "name": "designer-01",
  "vmRole": "Designer"
}
```

Creates connector record, returns the connector token (the only time it is sent
in plaintext - user copies it into their config file or Claude agent does it).

Response:

```json
{
  "id": "conn_a3f9x2k1",
  "name": "designer-01",
  "vmRole": "Designer",
  "token": "tok_7f2a9x...",
  "configYaml": "relay_url: wss://yourservice.com\nrelay_token: tok_7f2a9x...\n..."
}
```

The token is the connector ID stored in the database. It is not encrypted -
it is a random UUID that functions as a bearer token. If compromised, user
deletes the connector and creates a new one.

```
GET  /api/connectors          List user's connectors with online/offline status
DELETE /api/connectors/:id    Delete connector (disconnects active session)
```

### Config File Generation

```
GET /api/connectors/:id/config
```

Returns a ready-to-use `.yaml` config file for the VM connector.

File content:

```yaml
# Claude WebUI Connector Config
# Generated: 2026-04-09
# Place this file at ~/claude-webui/.env on your VM
# Then run: systemctl restart claude-webui

RELAY_URL=wss://yourservice.com/relay
RELAY_TOKEN=tok_7f2a9x...

# These values were set at registration - edit if needed
VM_ROLE=Designer
VM_NAME=designer-01
```

The config file is actually a shell-compatible env file. The VM's `.env` file
gets these values merged in.

## WebSocket Relay Server

### Connector Connection Lifecycle

```
VM Connector connects to: wss://yourservice.com/connector

1. WebSocket connection opens
2. Connector sends: { "type": "auth", "token": "tok_7f2a9x..." }
3. Server validates token against database
4. On valid: { "type": "auth_ok", "connectorId": "conn_a3f9x2k1" }
   - Channel manager registers this WebSocket as the live channel for this connector
   - Update last_connected timestamp in database
   On invalid: { "type": "auth_error", "message": "Invalid token" }, close connection
5. Server sends ping every CONNECTOR_PING_INTERVAL ms
6. Connector must respond with pong within CONNECTOR_TIMEOUT ms
7. On disconnect: channel manager marks connector offline, attempts pending requests fail with 503
```

### Channel Manager (`relay/channel-manager.ts`)

In-memory map:

```typescript
Map<connectorId: string, {
  ws: WebSocket,
  userId: string,
  pendingRequests: Map<requestId: string, {
    resolve: (response) => void,
    reject: (error) => void,
    timeout: NodeJS.Timeout
  }>
}>
```

**Methods:**

- `register(connectorId, userId, ws)` - add live connector
- `deregister(connectorId)` - remove on disconnect
- `isOnline(connectorId)` - check if connector is connected
- `sendRequest(connectorId, request)` - send HTTP request, return Promise of response
- `getOnlineConnectors(userId)` - list connected connectors for a user

Request timeout: 60 seconds. If connector does not respond in time, reject with 504.

## HTTP Proxy Layer

### Browser-to-Connector Request Flow

Browser requests arrive at:

```
/vm/:connectorId/*
```

Example: `/vm/conn_a3f9x2k1/api/files/tree?path=/projects/alpha`

**Proxy handler (`handlers/proxy.ts`):**

```
1. Validate session cookie - must be authenticated
2. Validate connectorId belongs to the authenticated user
3. Check channel manager - is this connector online?
   If offline: return 503 { "error": "VM is offline", "connectorName": "designer-01" }
4. Extract path: everything after /vm/:connectorId/ → /api/files/tree
5. Build request message:
   {
     "type": "http_request",
     "requestId": "<uuid>",
     "method": "GET",
     "path": "/api/files/tree",
     "query": "path=/projects/alpha",
     "headers": { forwarded headers, strip cookie },
     "body": null
   }
6. Send via channel manager, await response (60s timeout)
7. Forward response back to browser with original status and headers
```

**Streaming responses (chat):**

For streaming NDJSON (Claude chat responses), the proxy must stream chunks as
they arrive rather than buffering. Use Server-Sent Events or chunked transfer
encoding.

```
1. Same auth and routing as above
2. Set response headers: Content-Type: text/event-stream, Transfer-Encoding: chunked
3. Each stream_chunk message from connector: write chunk to response immediately
4. On stream_end: close response
5. On browser disconnect: send abort message to connector
```

**WebSocket proxying (if needed for future):**
Not required in Phase 2 - all real-time is handled via NDJSON streaming over HTTP.

## API Reference for Agent-Driven Setup

Agents need to be able to set up a VM without human intervention.
The relay exposes an API key mechanism for programmatic access (separate from
browser session cookies).

```
POST /api/agent-keys          Create an API key for agent use
GET  /api/agent-keys          List user's API keys
DELETE /api/agent-keys/:id    Revoke an API key
```

API key is passed as: `Authorization: Bearer <key>`

With an API key, agents can:

- `POST /api/connectors` - register a new VM
- `GET /api/connectors/:id/config` - download config file
- `GET /api/connectors` - check connector status

The user creates one API key from the dashboard, gives it to their Claude agent.
The agent handles everything else.

## Security Model

| Concern                      | Mitigation                                                                       |
| ---------------------------- | -------------------------------------------------------------------------------- |
| Stolen connector token       | Token only routes traffic to that VM; delete connector to revoke                 |
| User A accessing User B's VM | connectorId validated against session's userId on every request                  |
| Path traversal via proxy     | VM-side validates all paths (Phase 1 responsibility)                             |
| DDoS on relay                | Rate limiting: 100 req/min per session, 10 concurrent streams per user           |
| Relay server compromise      | No API keys, no file contents stored; attacker gains routing config only         |
| Session hijacking            | httpOnly cookies, secure flag, SameSite strict                                   |
| Connector token brute force  | Tokens are UUIDs (122 bits of entropy); rate limit auth attempts to 5/min per IP |

---

## Rate Limiting

Apply to all routes via middleware:

- Unauthenticated routes: 20 req/min per IP
- Authenticated API routes: 100 req/min per user
- Proxy routes: 200 req/min per user
- OAuth callback: 10 req/min per IP
- Connector auth: 5 attempts/min per IP

## Deployment

The relay server is a single Node.js process. Deployment target is any VPS
(Hetzner, DigitalOcean, Fly.io, etc.).

**Recommended minimum spec:** 1 vCPU, 512MB RAM, 10GB disk

At this spec, the relay can handle approximately 500 simultaneous connected
VM connectors and 100 concurrent browser sessions.

**Process management:** systemd service (same pattern as Phase 1)

**Reverse proxy:** Caddy in front of the Node.js process for:

- Automatic HTTPS (Let's Encrypt)
- WebSocket proxying (for connector connections)
- Compression

**Caddy config:**

```
yourservice.com {
  reverse_proxy localhost:3000
}
```

Caddy handles WebSocket upgrade headers automatically.

## What The Relay Does NOT Do

To be explicit for the builder:

- Does NOT store conversation history (that lives on the VM)
- Does NOT store project files (those live on the VM)
- Does NOT store or handle Anthropic API keys (those live on the VM)
- Does NOT run Claude (that runs on the VM)
- Does NOT inspect the content of proxied requests beyond routing headers
- Does NOT modify proxied request or response bodies
